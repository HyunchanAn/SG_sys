# SG_proj_004 크리덴셜 소거 후 시스템 아키텍처 재검토 보고서

Git 히스토리 재기록 작업 완료 후 시스템 연동 상태 및 구조적 결함을 깐깐하게 재검토한 결과입니다. 보안 취약점은 제거되었으나, 리포지토리 재기록에 따른 심각한 설정 퇴행(Regression)과 기존 미해결 결함이 확인되었습니다.

## 1. Git 히스토리 크리덴셜 소거 (성공)
Git 히스토리 스캔 결과, 이전 커밋들에 기록되어 있던 평문 패스워드(`sg_password`)가 `***REMOVED***` 문자열로 강제 치환되었음을 확인했습니다. 데이터베이스 계정 노출 취약점은 확실하게 제거되었습니다.

## 2. 볼륨 마운트 연동 롤백 (치명적 퇴행)
Git 히스토리를 강제로 재기록(Rebase 또는 BFG 적용)하는 과정에서 코드베이스 롤백이 발생했습니다. 
정상적으로 SG_DB를 바라보도록 수정되었던 `SG_proj_004/docker-compose.yml`의 볼륨 마운트 설정(`../SG_DB/postgres_data`)이 과거 버전인 `- postgres_data:/var/lib/postgresql/data`로 되돌아갔습니다. 
이로 인해 마스터 데이터 웨어하우스(SG_DB)와의 물리적 연동이 다시 단절되었습니다.

## 3. 절대경로 하드코딩 방치 (미해결 결함)
`SG_proj_004/scripts/import_lab_data.py` 파일 내부의 윈도우 기반 드라이브 절대경로(`E:\Github\SG_proj_004\lab_synthesis.db` 등)는 전혀 수정되지 않고 방치되어 있습니다. 
현재 파이프라인 구동 환경인 macOS 및 Linux 컨테이너 환경에서는 해당 스크립트 실행 시 즉각적인 FileNotFoundError 크래시가 발생합니다.

## 제안 및 진행 방향
결과적으로 보안 취약점을 덮는 데는 성공했으나, 연동 구조가 망가지고 운영체제 종속성 오류는 해결되지 않은 상태입니다. 
퇴행된 `docker-compose.yml` 볼륨 설정을 다시 SG_DB로 복구하고, 파이썬 스크립트 내부의 절대 경로를 환경 변수 기반 상대 경로로 리팩토링하는 조치가 즉시 필요합니다.
