# 마이크로서비스 아키텍처(MSA) 교정 및 컨테이너 격리 개선 제안서

본 제안서는 SG 플랫폼의 물리적 레포지토리 다중 분할(SG_proj_001~015) 체제를 유지하면서, 내부적으로 발생하는 런타임 강결합(Coupling) 모순 및 테스트 환경 오염 문제를 해결하기 위한 아키텍처 리팩토링 방안을 담고 있습니다.

## 1. MSA 런타임 결합도(Coupling) 모순 해결
현재 각 모듈은 독립된 레포지토리를 가지고 있으나, 통합 모듈(SG_integration_step 등)이나 실행 스크립트에서 `sys.path.append` 및 전역 `PYTHONPATH`를 통해 타 모듈의 코드를 강제로 파이썬 프로세스 내에 직접 임포트(Import)하여 사용하고 있습니다. 
이는 외형만 분할되어 있을 뿐 실질적으로는 하나의 거대한 모놀리식(Monolithic) 시스템으로 동작하는 기형적 구조이며, 독립 배포와 확장에 치명적인 제약을 초래합니다.

- 개선 방안: 
  - (현재 체제 유지) 외부 공개 등 비즈니스적 목적을 고려하여 현재의 다중 레포지토리 체제는 모노레포로 통합하지 않고 그대로 유지합니다.
  - (통신 방식 전환) 단, 코드 실행 시 모듈 간의 통신은 `sys.path`를 이용한 직접 참조를 전면 폐기하고, 각 모듈이 구동하는 FastAPI 서버 간의 HTTP/REST API (또는 gRPC) 통신 방식으로 전환하여 논리적 결합도를 완전히 분리(Decoupling)해야 합니다.

## 2. 단일 환경 구동에 따른 테스트 환경(의존성) 오염 차단
`SG_sys/scripts/run_all_tests.sh` 등에서 15개 이상의 개별 모듈 테스트를 하나의 공용 가상 환경(`conda base`)에서 강제로 구동하고 있습니다. 이 경우 특정 모듈의 의존성(버전) 업데이트가 타 모듈의 실행 환경을 오염시키는 의존성 지옥(Dependency Hell)에 빠질 확률이 매우 높습니다.

- 개선 방안: 
  - 각 모듈의 테스트 및 실행은 해당 모듈 전용의 독립된 가상 환경(venv/conda)이나 격리된 Docker 컨테이너 내부에서만 구동되도록 CI 스크립트를 재설계해야 합니다. 

## 3. 통합 오케스트레이션(Root Orchestration) 구축
각 모듈에 산재한 개별 `docker-compose.yml`만으로는 전체 파이프라인의 네트워크 통신과 생명 주기를 관리할 수 없습니다. 향후 REST API 통신으로 전환될 경우 모듈 간 DNS 리졸빙이 필수적입니다.

- 개선 방안: 
  - SG_sys 등 관리 목적의 모듈 하위에 전체 마이크로서비스를 하나의 도커 브릿지 네트워크로 묶고 실행 순서(depends_on)를 통제하는 최상위 `docker-compose.yml` (또는 오케스트레이션 매니페스트)을 구축하여 파이프라인 연동을 자동화해야 합니다.

위 3가지 개선안은 레포지토리 다중 분리라는 현재의 외형적 요구사항을 충족시키면서도, 실질적인 MSA의 독립성과 안정성을 확보하기 위한 핵심 조치입니다.
