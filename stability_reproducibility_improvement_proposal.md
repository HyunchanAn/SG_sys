# 시스템 안정성 및 모델 재현성 개선 제안서

본 제안서는 SG 플랫폼(SG_proj_001, SG_proj_004, SG_sys 등)의 시스템 안정성과 머신러닝 모델의 재현 가능성(Reproducibility)을 위협하는 3가지 치명적 결함을 식별하고 이를 해결하기 위해 작성되었습니다.

## 1. 의존성 언피닝(Unpinning)에 따른 재현성 파괴 위험
현재 SG_proj_001 및 SG_proj_004 모듈의 `pyproject.toml`과 `requirements.txt`에는 `numpy`, `scikit-learn`, `xgboost`, `pandas` 등 핵심 머신러닝 패키지의 버전이 명시되어 있지 않거나 하한선(>=)으로만 설정되어 있습니다.
의존성 버전을 고정(Pinning)하지 않으면 추후 패키지가 업데이트될 때 모델 가중치 호환성이 깨지거나 내부 알고리즘 로직이 변경되어 과거의 실험 결과를 완전히 재현할 수 없게 됩니다.

- 개선 방안: 모든 요구사항 패키지의 버전을 `==` 기호를 사용하여 현재 작동이 검증된 환경과 동일하게 프리징(Freezing) 처리해야 합니다.

## 2. 글로벌 난수 시드(Random Seed) 고정 누락
AI 파이프라인의 핵심인 SG_proj_001 코드를 전수 검사한 결과, `numpy.random.seed()`나 `torch.manual_seed()`와 같은 글로벌 난수 통제 로직이 전무합니다. 
데이터 분할이나 모델 가중치 초기화 과정에 시스템 난수가 무작위로 개입되면 코드를 구동할 때마다 모델 성능 지표가 요동치므로 재현성을 보장할 수 없습니다.

- 개선 방안: 모듈 실행의 최상단 진입점(Entry point)에 글로벌 난수 시드를 강제로 고정하는 설정을 추가하여 모델의 결정론적(Deterministic) 동작을 보장해야 합니다.

## 3. 하드웨어 연산 리소스 강제 스로틀링
SG_sys/scripts/run_all_tests.sh 파일 상단에 `OMP_NUM_THREADS=1` 및 `MKL_NUM_THREADS=1`이 선언되어 있어 멀티스레딩이 원천 봉쇄되고 있습니다.
테스트 환경의 프로세스 충돌을 방지할 목적이라도, 이 환경 변수 설정이 프로덕션 컨테이너나 병렬 처리 환경으로 파급될 경우 현재의 고성능 멀티코어 리소스(12코어 24스레드 등)를 전혀 활용하지 못하는 심각한 병목 현상을 유발합니다.

- 개선 방안: 멀티스레딩 억제 환경 변수는 테스트 파이프라인 내부 로직에만 격리하여 적용하고, 실제 모델 학습 및 인퍼런스 서버 구동 시에는 해당 변수 주입을 제거해야 합니다.

위 세 가지 결함은 시스템 벤치마크와 프로덕션 안정성에 직결되므로 즉각적인 설정 리팩토링을 제안합니다.
