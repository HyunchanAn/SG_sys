# 작업지시서: 공통 API 스키마 패키지 통합 (API Schema Unification)

본 작업지시서는 기존 모놀리식 모듈 참조 방식(`sys.path`)에서 REST API 기반의 MSA 환경으로 성공적으로 전환됨에 따라, 런타임에 발생할 수 있는 데이터 타입 불일치(Type Mismatch) 및 스키마 파편화 리스크를 방지하기 위해 작성되었습니다.

## 1. 배경 및 목적
- **현상**: 여러 레포지토리(001~015)가 완전히 분리된 채 REST API로 통신하고 있으나, 서로 주고받는 데이터의 Pydantic 스키마가 각 모듈 내부에 독립적으로 정의되어 있습니다.
- **리스크**: 특정 모듈의 데이터 필드명이 변경되거나 타입이 바뀔 경우, 이를 의존하는 다른 모듈들은 런타임에 통신 오류(422 Unprocessable Entity)가 발생할 때까지 장애를 인지할 수 없습니다. 모노레포로 코드를 합치지 않고 15개의 독립 레포지토리를 유지하기로 한 아키텍처 원칙 하에서는 이 위험이 더욱 치명적입니다.
- **목표**: 모든 시스템 간 통신에 쓰이는 데이터 모델(DTO, Pydantic)을 단일 공통 라이브러리(Single Source of Truth)로 묶어 컴파일/정적 검사 단계에서 에러를 조기 검출합니다.

## 2. 작업 스코프 및 절차

### 단계 1: 공통 스키마 내장 라이브러리 구축 (SG_sys 활용)
- 레포지토리를 새로 생성하지 않고, 관리용 루트인 `SG_sys` 디렉토리 내부에 `shared_schemas` 패키지 폴더를 신설합니다.
- 해당 라이브러리는 비즈니스 로직(실제 연산 함수)을 일절 포함하지 않으며, 오로지 Pydantic 베이스 모델, Enum 상수, DTO 클래스만을 갖습니다.

### 단계 2: 파편화된 데이터 모델 추출 및 마이그레이션
- 현재 각 단위 모듈(`SG_proj_001`, `002`, `004` 등) 내부에 흩어져 있는 API Request / Response 규격 모델들을 찾아내어 `SG_sys/shared_schemas`로 이동시킵니다.
- (예) 표면 SFE 데이터, 3D 곡률 분석 모델, 대체재 매칭 TOPSIS 스코어 규격 등

### 단계 3: 의존성 패키징 및 주입
- `SG_sys/shared_schemas` 폴더 내부에 `setup.py`나 `pyproject.toml`을 구성하여 로컬 파이썬 패키지로 만듭니다.
- `SG_sys/docker-compose.yml`에서 각 컨테이너의 빌드 컨텍스트(Build Context)를 상위로 조정하거나 볼륨 마운트를 통해, 도커 빌드 시 해당 패키지가 공통 의존성으로 자동 설치되도록 파이프라인을 수정합니다.

### 단계 4: 로컬 코드 리팩토링 및 런타임 검증
- 모든 모듈의 코드에서 `from .schemas import SurfaceData` 등의 로컬 참조를 지우고, `from shared_schemas.sfe import SurfaceData` 형태로 공통 패키지를 바라보도록 수정합니다.
- 리팩토링 후 `run_all_tests.sh`를 구동하여 데이터 직렬화/역직렬화(Serialization)가 모두 정상적으로 작동하는지 통합 검증합니다.

## 3. 오너 게이트 및 승인 사항
- [x] `SG_sys` 내부 구조 변경 및 전체 컨테이너 도커 빌드 컨텍스트(`docker-compose.yml`) 대규모 수정 승인.
- [x] 통합 스키마 런타임 적용 시 기존 모듈 간 하위 호환성 단절(Breaking Changes) 허용.

3. **코드 리팩토링 및 1차 검증**: 각 하위 모듈에서 하드코딩되었던 스키마를 걷어내고 `shared_schemas`를 바라보도록 모두 수정 후 문법 오류/임포트 검증을 완료했습니다.

**[Next Steps 및 최종 판단 (2026-07-19 수정)]**
로컬 환경의 빌드 병목을 줄이고 CI를 가속하기 위해 **Windows 메인 워크스테이션에서 15개 모듈 전체 리빌딩 및 GPU 텐서 가속 검증을 시도**했습니다. 
그러나 윈도우 도커 엔진(Docker Desktop) 환경에서 원격 네트워크 드라이브(UNC 경로)를 로컬 볼륨으로 직접 바인딩(`../SG_DB/postgres_data`)할 때 치명적인 호환성 오류(`is not a valid Windows path`)가 발생했습니다.

이를 윈도우에서 우회하기 위해 도커 전용 네임드 볼륨(Named Volume) 체계로 리팩토링할 경우, 기존 맥북에 보존되어 있던 수 GB 단위의 귀중한 데이터베이스 내용이 빈 통으로 포맷되는 리스크가 확인되었습니다.
이에 따라 **시스템 데이터의 영속성 보존과 무결성을 최우선 원칙으로 판단**하여, 윈도우에서의 무리한 GPU 테스트를 취소하고 본래의 **맥북 환경 로컬 도커 엔진에서 파이프라인 구동을 재개(회귀)하는 것으로 최종 인계**하였습니다. 맥북 측 터미널에서 `docker-compose up -d --build`를 안전하게 구동하십시오.
